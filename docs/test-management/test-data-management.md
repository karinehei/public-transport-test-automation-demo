# Test Data Management – Public Transport Ticketing System

How test data is created, maintained, and kept isolated for API, UI, and E2E automation.

## Overview

The ticketing system uses **in-memory storage** for tickets. No persistent database is used in test environments, so:

- Each API restart clears all tickets
- Tests create their own data (zones, tickets) during execution
- No shared test data files or seed data required

## Test Data Creation

### API Tests

| Test | Data Created | Method |
|------|--------------|--------|
| Get Zones | None (read-only) | GET /zones |
| Purchase Ticket | 1 ticket | POST /tickets |
| Fetch Ticket | 1 ticket | POST /tickets, then GET |
| Validate Ticket | 1 ticket, then validated | POST /tickets, POST /validate |

**Pattern**: Each test creates the data it needs via API calls. No pre-seeded data.

### UI Tests

| Test | Data Created | Method |
|------|--------------|--------|
| Complete Ticket Purchase And Validation UI Flow | 1 ticket | UI: select zone, click Buy Ticket |

**Pattern**: UI interaction creates ticket via API (frontend calls backend).

### E2E Tests

| Test | Data Created | Method |
|------|--------------|--------|
| Complete Ticket User Journey | 1 ticket | API: GET zones, POST /tickets, GET /tickets, POST /validate |

**Pattern**: Full flow creates and consumes data in one test.

## Data Isolation

- **No shared state**: Each test creates its own tickets
- **No cleanup required**: In-memory store is reset when API restarts
- **Parallel safety**: Each pipeline job runs in a separate container/agent; no cross-job data

## Repeatability

- **Zones**: Fixed set (AB, ABC, ABCD, BC, CD) – no configuration needed
- **Tickets**: Generated by API with UUID; no conflicts
- **Validation**: One-time; second validation returns "already used"

## Test Data for API/UI/E2E Automation

| Test Type | Data Source | Isolation |
|-----------|-------------|-----------|
| API | Created in test via RequestsLibrary | Per test case |
| UI | Created via UI (API call from frontend) | Per test case |
| E2E | Created in test via RequestsLibrary | Per test case |

## Extending for Persistent Storage

If the system gains a database:

1. **Option A – Reset per run**: Run migrations or truncate before each pipeline run
2. **Option B – Test database**: Dedicated DB per run; Docker Compose with temp volume
3. **Option C – Seed data**: Minimal seed (e.g. zones); tests create tickets
4. **Option D – Transactions**: Use DB transactions; rollback after each test (if supported)

## Best Practices

- Create data in test; avoid external seed files when possible
- Use unique identifiers (UUID) to avoid collisions
- Keep tests independent; no test should depend on another’s data
- Document any required setup (e.g. zones) in test plan and keywords
